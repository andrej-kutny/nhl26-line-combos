\section{Methodology}

This project primarily follows \textbf{Tracks 2 + 3} (ASP-based reasoning), with a focus on combinatorial optimization using Answer Set Programming. The system architecture separates data management, ASP rule encoding, solver integration, and result persistence into distinct layers. The implementation covers both \textbf{Goal 1} (batch pipeline optimization) and \textbf{Goal 2} (interactive optimization), with Goal 2 integrated at the Core/ASP level.

\subsection{System Architecture}

The system follows a layered architecture with clear separation of concerns:

\begin{itemize}
    \item \textbf{Presentation Layer}: React-based frontend for user interaction
    \item \textbf{API Layer}: FastAPI backend providing REST endpoints
    \item \textbf{Core Layer}: Data models, SQLite data loading, and business logic
    \item \textbf{ASP Layer}: Clingo integration for optimization (Goal 1 and Goal 2)
    \item \textbf{Data Layer}: SQLite database storing player cards and line combinations
\end{itemize}

This architecture enables parallel development: the backend provides interfaces and orchestration while ASP encodings are developed and tested independently.

\subsection{Data Model and Storage}

The system uses \textbf{SQLite} for persistent storage, populated from CSV files via an automated ingestion pipeline. The database schema includes:

\begin{itemize}
    \item \textbf{Player Tables}: \texttt{forwards}, \texttt{defense}, \texttt{goalies} - storing card attributes (OVR, salary, team, nationality, event)
    \item \textbf{Combo Tables}: \texttt{forward\_combos}, \texttt{defense\_combos} - storing line combination templates
    \item \textbf{Result Tables (Goal 1)}: \texttt{goal1\_runs}, \texttt{goal1\_stage\_a\_results}, \texttt{goal1\_concrete\_lines} - persisting batch optimization results
\end{itemize}

The data loader (\texttt{DataLoader}) provides efficient querying with filtering by team, nationality, event, and OVR ranges. This design allows the ASP layer to work with pre-filtered, structured data rather than raw CSV files.

\subsection{ASP Encoding for Goal 1}

Goal 1 implements a \textbf{two-stage optimization pipeline} to handle the combinatorial explosion:

\subsubsection{Stage A: Abstract Optimization}

Stage A optimizes over \textit{line combination templates} without considering specific players. The ASP encoding includes:

\textbf{Main Predicates:}
\begin{itemize}
    \item \texttt{forward\_combo(ID, Reward, Type, Entry1, Entry2, Entry3)} - Forward combo definitions
    \item \texttt{defense\_combo(ID, Reward, Type, Entry1, Entry2)} - Defense combo definitions
    \item \texttt{player\_attr(PlayerSlot, Attribute)} - Abstract attribute assignments to player slots
    \item \texttt{fwd\_active\_combo(ID, Reward, Type)} / \texttt{def\_active\_combo(ID, Reward, Type)} - Activated combos
    \item \texttt{total\_reward(Total)} - Weighted sum of activated combo rewards
\end{itemize}

\textbf{Key Rules:}
\begin{itemize}
    \item \textbf{Attribute Generation}: Each player slot (1-3 for forwards, 1-2 for defense) can have at most one attribute per type (team, nationality, event)
    \item \textbf{Combo Activation}: A combo activates when player slots match its requirements in canonical order
    \item \textbf{Symmetry Breaking}: Lexicographic ordering of attributes prevents redundant solutions
    \item \textbf{Optimization}: \texttt{\#maximize \{ T : total\_reward(T) \}} maximizes weighted reward
\end{itemize}

\subsubsection{Stage B: Concrete Line Enumeration}

\texttt{ClingoStageBSolver} enumerates concrete player lines that satisfy Stage A combo selections. The ASP encoding includes:

\textbf{Main Predicates:}
\begin{itemize}
    \item \texttt{player(CardID, OVR, Team, Nationality, Event)} - Player card facts
    \item \texttt{card\_player(CardID, PlayerID)} - Mapping cards to real players (for uniqueness)
    \item \texttt{salary(CardID, Amount)} / \texttt{ap(CardID, Amount)} - Resource constraints
    \item \texttt{in\_line(CardID)} - Selected cards forming a line
    \item \texttt{combo\_satisfied(ComboID)} - Combo activation verification
\end{itemize}

\textbf{Key Rules:}
\begin{itemize}
    \item \textbf{Line Formation}: Exactly 3 players for forwards, 2 for defense
    \item \textbf{Uniqueness Constraint}: No duplicate \texttt{player\_id} within a line
    \item \textbf{Symmetry Breaking}: Canonical ordering by \texttt{card\_id} prevents duplicate lines
    \item \textbf{Combo Satisfaction}: All selected combos from Stage A must be satisfied by the line
    \item \textbf{Resource Constraints}: Salary cap and AP limits enforced, with combo bonuses providing relief
    \item \textbf{Optimization}: Multi-level optimization prioritizing combo bonuses, then OVR, then resource efficiency
\end{itemize}

The solver uses Clingo's \texttt{--opt-mode=optN} to enumerate all optimal solutions and includes deduplication logic to handle Clingo's progressive optimization output.

\subsection{Goal 1 Pipeline Implementation}

The pipeline orchestrator (\texttt{Goal1Pipeline}) coordinates:

\begin{enumerate}
    \item \textbf{Stage A Input Generation}: Converts SQLite combo data into ASP facts
    \item \textbf{Stage A Solving}: Executes rules (\texttt{rules.lp}, \texttt{fwd\_rules.lp}, \texttt{def\_rules.lp}) and parses \texttt{StageASolution}
    \item \textbf{Stage B Input Generation}: Builds candidate pools from the Stage A attribute keys
    \item \textbf{Stage B Solving}: Executes rules (\texttt{base.lp}, \texttt{forward\_line.lp}, \texttt{defense\_pair.lp}) and enumerates concrete lines
    \item \textbf{Result Storage}: Persists Stage A and Stage B results to SQLite
\end{enumerate}

The pipeline supports multiple optimization modes: \texttt{ovr}, \texttt{sal}, \texttt{ap}, \texttt{ovr\_sal}, and \texttt{ovr\_sal\_ap}, with configurable weights per position type.

\subsection{ASP Encoding for Goal 2 (Interactive Optimization)}

Goal 2 differs from Goal 1: it performs \textbf{direct concrete optimization} in one stage, selecting real player cards directly under user-provided constraints and optional combo requirements.

\subsubsection{Goal 2 Architecture}

Goal 2 follows a replicated pattern from Goal 1 (generator + solver + pipeline), but without the abstraction stage:

\begin{itemize}
    \item \textbf{Input Generation}: \texttt{Goal2InputGenerator} converts candidate players and combo definitions into ASP facts and computes weights for the target mode.
    \item \textbf{Solving}: \texttt{ClingoGoal2Solver} loads the Goal 2 ASP rule files from \texttt{src/asp/g2/}, runs Clingo, and parses models into concrete line results.
    \item \textbf{Orchestration}: \texttt{Goal2Pipeline} coordinates generator $\rightarrow$ solver and records timing/metadata.
\end{itemize}

Goal 2 supports:
\begin{itemize}
    \item \textbf{Positions}: forward (3 cards), defense (2 cards)
    \item \textbf{Optimization targets}: \texttt{ovr}, \texttt{sal}, \texttt{ap}, \texttt{ovr\_sal}, \texttt{ovr\_sal\_ap}
    \item \textbf{Optional combo enforcement}: required combo IDs can be enforced as hard constraints
\end{itemize}

\subsubsection{Goal 2 ASP Rules Organization}

The ASP rules for Goal 2 are modularized:

\begin{itemize}
    \item \texttt{g2/common.lp}: shared predicates for player attribute extraction and combo matching
    \item \texttt{g2/fwd\_main.lp} and \texttt{g2/def\_main.lp}: line formation and symmetry breaking
    \item \texttt{g2/*\_description.lp}: objective and scoring rules per target (OVR, SAL, AP)
\end{itemize}

Compared to Goal 1, Goal 2 is designed for interactive use and generally runs faster because it avoids the Stage A abstraction.

\subsection{ASP Rules Testing}

ASP and integration testing are performed at multiple levels:

\textbf{Goal 1 Tests}: validate Stage A rules and Stage B grounding, plus backend parsing and candidate generation.

\textbf{Goal 2 Tests}: include unit tests and integration tests for:
\begin{itemize}
    \item Weight calculation for all optimization targets
    \item Fact generation correctness for players and combos
    \item Model parsing correctness for forward and defense lines
    \item Pipeline execution and edge-case handling
    \item Optional rule-level validation when Clingo is installed
\end{itemize}

This multi-level test strategy validates ASP logic correctness and Python integration separately.

\subsection{Frontend Implementation}

The frontend is implemented as a React 19 application using TypeScript, Ant Design components, and React Router for navigation. The application provides a user interface for exploring players, viewing optimization results, and configuring optimization requests.

\textbf{Implemented Pages:}
\begin{itemize}
    \item \textbf{HomePage}: dataset statistics and navigation
    \item \textbf{PlayersPage}: player browsing and filtering
    \item \textbf{BestLinesPage}: Goal 1 results browsing by position and mode
    \item \textbf{OptimizePage}: interactive optimization form (Goal 2 integration pending in API layer)
\end{itemize}

\subsection{Pipeline Manager}

To ensure the frontend always has results to display, a \textbf{Pipeline Manager} module automatically runs the Goal 1 pipeline when results are missing. It provides:
\begin{itemize}
    \item \textbf{Result Checking}: \texttt{has\_results()}
    \item \textbf{Auto-Run}: \texttt{ensure\_results()}
    \item \textbf{Non-Blocking Execution}: executes in a thread pool using \texttt{asyncio.to\_thread()}
\end{itemize}

\subsection{API Endpoints}

The system exposes REST endpoints for:

\begin{itemize}
    \item \textbf{Player Queries}: \texttt{GET /players/\{type\}}
    \item \textbf{Combo Queries}: \texttt{GET /combos/\{type\}}
    \item \textbf{Best Lines (Goal 1)}: \texttt{GET /best/\{pos\}/\{mode\}} with auto-run pipeline
    \item \textbf{Pipeline Generation}: \texttt{POST /best/\{pos\}/\{mode\}/generate}
    \item \textbf{Interactive Optimization (Goal 2)}: implemented at Core/ASP level; API/frontend integration is a planned extension
\end{itemize}
