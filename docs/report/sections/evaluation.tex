\section{Evaluation}

We evaluated the system through multiple dimensions: correctness of ASP rules, pipeline functionality, data integrity, and backend integration behavior. Evaluation includes both Goal 1 (two-stage batch pipeline) and Goal 2 (direct interactive optimization), with Goal 2 currently evaluated at the Core/ASP integration level.

\subsection{ASP Rules Validation (Goal 1)}

The ASP rules for Stage A (abstract optimization) are validated through a comprehensive test suite using Clingo directly. Tests verify:

\begin{itemize}
    \item \textbf{Satisfiability}: all rule combinations (forward/defense $\times$ OVR/SAL/AP) produce satisfiable models
    \item \textbf{Optimization Correctness}: cost minimization returns expected optimal solutions
    \item \textbf{Weight Application}: multiplier constants correctly scale reward types
    \item \textbf{Threshold Lookup}: constraint-based filtering works as expected
    \item \textbf{No Duplicates}: solutions contain unique combo sets (symmetry breaking effective)
\end{itemize}

\textbf{Test Results}: all Stage A rule tests pass, confirming logical correctness of the encoding.

\subsection{Pipeline Infrastructure Testing (Goal 1)}

The Goal 1 pipeline infrastructure is tested end-to-end:

\begin{itemize}
    \item \textbf{Stage A Input Generation}: converts SQLite combo data to ASP facts
    \item \textbf{Stage A Solving}: Clingo execution and parsing into \texttt{StageASolution}
    \item \textbf{Stage B Input Generation}: candidate pools built from combo attribute keys
    \item \textbf{Stage B Solving}: concrete line enumeration with constraints and deduplication
    \item \textbf{Result Storage}: persisted results are retrievable via the API results endpoint
\end{itemize}

\subsection{Goal 2 Solver and Pipeline Testing}

Goal 2 is evaluated using a dedicated test suite covering unit-level and integration-level behaviors:

\begin{itemize}
    \item \textbf{Input generation}: player and combo facts are correctly formatted for the Goal 2 ASP rules
    \item \textbf{Weight calculation}: all optimization targets (\texttt{ovr}, \texttt{sal}, \texttt{ap}, \texttt{ovr\_sal}, \texttt{ovr\_sal\_ap}) compute correct weight vectors with position-dependent defaults
    \item \textbf{Model parsing}: forward and defense models are parsed into concrete line results with correct totals and activated combo IDs
    \item \textbf{Pipeline execution}: \texttt{Goal2Pipeline} correctly orchestrates generator $\rightarrow$ solver and returns timed results
    \item \textbf{Edge cases}: empty candidate lists, insufficient players, and missing combos are handled gracefully
\end{itemize}

These tests validate that Goal 2 is functional and consistent with the Goal 1 integration pattern at the backend level. Since the API/frontend endpoint wiring is not yet completed, evaluation focuses on correctness of the solver outputs and its supporting infrastructure.

\subsection{Data Integrity}

Data loading and storage are validated:

\begin{itemize}
    \item \textbf{CSV to SQLite Migration}: automated ingestion populates all tables
    \item \textbf{Data Completeness}: forwards, defense, and goalies are loaded with expected card counts
    \item \textbf{Referential Integrity}: player name lookups resolve correctly via ID tables
    \item \textbf{Query Performance}: SQLite indexes enable fast filtering (team, nationality, event, OVR ranges)
\end{itemize}

\subsection{Stage B Performance (Goal 1)}

\begin{itemize}
    \item \textbf{Enumeration Completeness}: Stage B enumerates all valid lines satisfying selected combos
    \item \textbf{Constraint Enforcement}: uniqueness, salary cap, and combo satisfaction are enforced
    \item \textbf{Optimization Quality}: optimization prioritizes combo rewards before secondary criteria
    \item \textbf{Deduplication}: distinct optimal line combinations are returned
\end{itemize}

\subsection{Pipeline Manager Performance}

Manual testing confirms that the pipeline manager preserves API responsiveness:

\begin{itemize}
    \item \textbf{Non-Blocking Execution}: Goal 1 pipeline runs in a thread pool
    \item \textbf{Automatic Result Generation}: first access triggers computation when missing
    \item \textbf{Concurrent Handling}: other endpoints remain responsive during pipeline execution
    \item \textbf{Error Resilience}: failures are logged without crashing the API
\end{itemize}

\subsection{Limitations of Current Evaluation}

While Goal 1 is evaluated end-to-end through API retrieval of stored results, Goal 2 has one remaining integration gap:

\begin{itemize}
    \item \textbf{Goal 2 API/frontend integration}: Goal 2 is implemented and tested at Core/ASP level, but the interactive HTTP endpoint wiring and frontend usage are not yet fully integrated, therefore end-to-end user-flow evaluation is pending.
    \item \textbf{Scalability}: full-scale performance profiling on large candidate pools is limited, although unit and integration tests cover typical pool sizes.
    \item \textbf{Goal 3 Full Team Optimization}: not implemented.
\end{itemize}

\subsection{Future Evaluation Plan}

Once the Goal 2 endpoint and frontend integration are completed, evaluation should include:

\begin{itemize}
    \item \textbf{End-to-End Interactive Quality}: validate that user constraints and combo requirements yield expected lines in the UI
    \item \textbf{Latency Measurements}: measure solve time across pool sizes and targets (interactive response budgets)
    \item \textbf{Correctness Verification}: validate that returned lines activate expected combos in user-visible output
    \item \textbf{Comparison Baselines}: compare against greedy heuristics or manual selection strategies
\end{itemize}
